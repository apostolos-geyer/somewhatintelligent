## 1. Context

This repo began as a template fork; per-client values were centralized in
`packages/config/src/deploy.ts` + `brand.ts` and substituted into
`*/wrangler.template.jsonc` by `scripts/render-wrangler.ts` (tokens like
`{{BASE_DOMAIN}}`, `{{D1_SPROUT_STAGING}}`), producing **gitignored**
`wrangler.jsonc` files. The repo is now permanently the Sprout platform — the
indirection buys nothing and costs:

- every fresh clone / CI run / agent session must run `bun run render` before
  wrangler, vite, vitest-pool-workers, or `wrangler types` work at all;
- the deployable config is not in git (unreviewable diffs, no blame);
- root scripts chain `render` into `dev`/`build`/`bootstrap`/`migrate`
  (root `package.json`), adding a serial step everywhere.

Decision (owner-confirmed): each worker gets **one checked-in `wrangler.jsonc`**
with **top-level = staging** and **`env.production`** as the only named env.
The current top-level "development" section disappears; local dev runs against
the staging config + `.dev.vars` overrides (see §5; the full dev story is
Spec 05's).

## 3. Target layout per worker

For each worker `W` with current template `W/wrangler.template.jsonc`:

1. Write a new **checked-in** `W/wrangler.jsonc` whose **top level is the
   current `env.staging` section merged over the shared/base keys** (main,
   compatibility_date/flags, account_id, observability, upload_source_maps,
   migrations, DO classes), and whose only `env` is `production` (current
   `env.production` content).
2. Delete `W/wrangler.template.jsonc`.
3. Substitute every `{{TOKEN}}` with its literal value from
   `packages/config/src/deploy.ts` / brand — the full token→value table is in
   §2 above and in `scripts/render-wrangler.ts:37`.

### 3.1 Worker names — the one footgun that would create new workers

Wrangler derives a named env's worker name as `<top-level name>-<envName>`
when the env block doesn't set `name`. Today: top-level `name` is e.g.
`sprout-guestlist` and `--env staging` deploys `sprout-guestlist-staging`.
After the flip:

- top-level `name` MUST be the literal staging name: `"sprout-guestlist-staging"`;
- `env.production` MUST set `name` explicitly: `"sprout-guestlist-production"`
  (otherwise wrangler would derive `sprout-guestlist-staging-production` — a
  brand-new worker, silently).

Apply to every worker. Grep the finished configs: every `env.production` block
must contain a `"name"` key; no config may contain a bare un-suffixed name.

### 3.2 Non-inherited keys checklist

Named envs inherit almost nothing. When flattening, verify **per worker** that
`env.production` re-declares everything it needs; nothing may rely on
inheritance from the (now staging) top level. Known per-env keys in this repo
(from the templates; the authoritative per-worker diff is in
[`02a-wrangler-binding-inventory.md`](02a-wrangler-binding-inventory.md)):

- `vars` (ENVIRONMENT, \*\_URL, BRAND_RESOLUTION, RTK_APP_ID, CF_ACCOUNT_ID, …)
- `d1_databases` (per-env database_id + database_name)
- `durable_objects.bindings` + `migrations` (DO migrations are per-env in the
  sense that each env block repeats them — keep the frozen `v1` tags exactly)
- `queues.producers` / `queues.consumers` (per-env queue names `…-staging`)
- `services` (per-env service names, **including `entrypoint: "Roadie"` +
  `props: { callerApp: … }` on every ROADIE binding** — dropping these breaks
  every roadie call; see `docs/runbooks/roadie-r2-provisioning.md`)
- `ai` / `vectorize` / `browser` (repeat per env; per-env index names)
- `analytics_engine_datasets`
- `routes` / custom domains / `workers_dev` / `preview_urls`
- `observability` (production has logs enabled + head_sampling_rate)
- R2 buckets (roadie), `send_email` (promoter, production-only), `ratelimits`
  (marketing, distinct namespace per env), `triggers.crons` (roadie) — whatever
  `02a` lists for that worker

Three traps from the deploy-log audit (full detail in `02a` §3):

- **`workers_dev` inverts**: bouncer/identity/sprout set `workers_dev: true`
  only in staging today; after the flip production would INHERIT `true`.
  `env.production` must set `"workers_dev": false` for those three. While
  here: set `"preview_urls": true` at top level (staging) and `false` in
  `env.production` for every worker — Spec 04 requires it.
- **`BNC_ATT_KID` is dev-only** — do NOT hoist it into staging/production vars
  (kit's stamper no-ops outside development). Dev gets it from seeded
  `.dev.vars`; add `BNC_ATT_KID=dev` to identity's seeder (sprout/bouncer
  already write it). This also silences the recurring deploy warning.
- **marketing `assets`** is top-level-only today — confirm the ASSETS binding
  survives for production via the §6 dry-run.

Special care — **bouncer**: it carries the zone routes. The staging-wildcard
shadowing incident (2026-07-01) means: the org wildcard route
(`*.sproutportal.ca/*`) must exist **only** in `env.production`, never at the
(staging) top level, or every `*-staging` custom domain 404s again. Check the
template's current comments before flattening bouncer, and preserve them.

## 5. Local dev consequences (bounded here, expanded in Spec 05)

Top-level config now carries staging vars (`ENVIRONMENT=staging`, staging URLs)
and staging resource names. Local dev correctness comes from two mechanisms:

1. **`.dev.vars` overrides** — wrangler dev and the Cloudflare Vite plugin load
   `.dev.vars` on top of top-level `vars`. The per-worker
   `scripts/bootstrap.ts` seeders already write `ENVIRONMENT=development`,
   local URLs, dev secrets. Verify every var the old top-level dev section
   defined is now covered by the seeded `.dev.vars` (diff old template
   top-level `vars` vs the seeder's output per worker; add missing keys to the
   seeders, e.g. sprout's `BRAND_RESOLUTION=subdomain`, `BNC_ATT_KID=dev`,
   `CF_ACCOUNT_ID`).
2. **Local sim ignores remote ids** — D1 local driver keys on `database_name`
   (a `…-staging-db` name locally is just a differently-named local sqlite
   file); queues/DO/AE are local sims; AI/Vectorize/Browser were already
   remote-proxied in dev.

Two things to re-verify by hand after the flip (they gate merge):

- **vitest-pool-workers**: each test vite.config reads
  `wrangler: { configPath: "./wrangler.jsonc" }` → tests now load staging
  top-level. Stub worker names in e.g. `services/guestlist/vite.config.ts`
  must match the service names the config references (now `sprout-promoter-staging`
  etc.) — update the `wp` prefix logic accordingly (it currently builds
  `sprout-promoter`). Run the full `bun run test` + `test:pool` tiers.
- **portless dev stack**: boot the stack per
  `docs/sprout/10-local-stack-and-testing-runbook.md` and walk sign-in →
  portal. Service-binding resolution in local dev registry uses the top-level
  worker names — all workers now register with `-staging` suffixed names, and
  cross-worker `services` entries at top level reference exactly those names,
  so the registry stays self-consistent. Confirm empirically.

## 6. Byte-equivalence gate (the actual safety proof)

Before deleting anything, on a scratch branch:

1. `bun run render` with the CURRENT system → save all generated
   `wrangler.jsonc` files aside (`/tmp/rendered-before/`).
2. Author the new static configs.
3. For each worker and each env, compare **effective config**, not file text:
   `wrangler deploy --dry-run --outdir /tmp/x` for staging
   (old: `--env staging`; new: no flag) and production (`--env production`,
   both) — and diff the resolved `name`, bindings, vars, routes between old and
   new. Any drift beyond the intended (dev section removed) is a bug.
   A lighter proxy: `jq`-normalize old rendered file's `env.staging`-merged view
   vs new top level. The dry-run route is authoritative because it exercises
   wrangler's own inheritance/derivation rules (esp. names, §3.1).
   **Sprout (TanStack-Start + CF vite plugin) bakes the target env into its
   generated `dist/server/wrangler.json` at BUILD time** — it reads
   `CLOUDFLARE_ENV` in `apps/sprout/vite.config.ts`. A bare `wrangler deploy
--dry-run --env production` against an on-disk `dist/` last built for staging
   hard-fails: `does not match the target environment "staging" … used when
building`. Rebuild for the env first — run sprout's own `deploy:<env>` build
   step (`CLOUDFLARE_ENV=production vp run build`) — then dry-run. The worker
   services (plain wrangler, no pre-build) dry-run directly; marketing (Astro)
   emits only the harmless `No environment found … "production"` warning (§02a
   §3.5), not a hard fail.
4. `bun run types` per worker + full typecheck + test tiers green.
5. First real deploy after merge: staging via the normal CI path, then check
   `wrangler deployments list` per worker shows an update to the SAME worker
   (no new worker names appeared on the account).

## 7. Acceptance criteria

- No `wrangler.template.jsonc`, no `scripts/render-wrangler.ts`, no `render`
  script, no `{{TOKEN}}` anywhere in the repo.
- `git grep -l 'wrangler.jsonc' -- .gitignore` → nothing (configs are tracked).
- Fresh clone + `bun install` + `bun run types` + `bun run typecheck` works
  with **zero** pre-steps (no render).
- Staging + production dry-run diffs are clean (§6.3).
- Dev stack boots; tests green; docs/CLAUDE.md updated.
