# 04 — Sprout UI: surfaces, screen by screen

> **Scope.** A close look at the Sprout product's UI — every one of the spec's
> surfaces — mapped onto greenroom's design system (`packages/ui` primitives,
> `packages/design` tokens). For each surface: desktop + mobile layout, the exact
> components used (existing primitive vs. build-new vs. variant), interaction +
> motion, empty/loading/error states, and accessibility. Then a single
> **component inventory** table, the **runtime theming model** from the UI side,
> and the form convention.
>
> This builds directly on the three foundation docs and reuses their names
> verbatim: the route map and shell mechanics from
> [03 — App Structure](./03-app-structure.md) (`_portal.tsx`, `LayerStack`,
> `RotatingHero`, `BannerRail`, `AiBubble`, `BrandStyle`, `BrandLogo`,
> `?section=` search param), the tables from
> [02 — Data Model](./02-data-model.md) (`brand_theme`, `portal_config`, `hero_slides`,
> `banner_cards`, `products`, `reviews`, `decks`, `deck_progress`, `assets`,
> `physical_requests`, `quizzes`, `attempts`, `posts`, `comments`, `chat_rooms`,

> [01 — Architecture](./01-architecture.md) (envelope-first auth, the
> single `GroupChatRoom` DO — group chat keyed `idFromName(brandId)`, feed
> comments keyed `idFromName(brandId:postId)` — roadie blobs, brand resolution).

## How this maps to the design system (read first)

Three system facts drive every surface below; they are not repeated per surface.

1. **`useAppForm` is mandatory for every form.** `useAppForm` from
   `@greenroom/ui/hooks/use-app-form` is the only sanctioned form pattern in the
   repo (project memory `feedback_repo_form_convention`: quiz Wave-3 hand-rolled
   forms were flagged and retrofitted). Every form on every surface — the
   request-physical form, the contact form, the access-request, the review
   composer, the comment box, all Brand-Admin editors, the AI booking note —
   uses `form.AppField` + the lazy field components (`TextField`, `EmailField`,
   `TextareaField`, `SelectField`, `CheckboxField`, `RadioField`, `SwitchField`,
   `SliderField`) and `form.AppForm` + `form.SubmitButton`. **Never** `useState`
   - native `<input>`. The fields confirmed on disk:
     `packages/ui/src/components/form/{text-field,email-field,textarea-field,select-field,checkbox-field,radio-field,switch-field,slider-field,submit-button,reset-button}.tsx`.

2. **Color/radius/material are token-driven, never literal.** Surfaces compose
   `surfaceMaterials.brutal` (the signature soft-elevation card,
   `packages/ui/src/lib/materials.ts:24`) + `Card` variants
   (`default|soft|neo|neo-inset|glass|dark`, `packages/ui/src/components/ui/card.tsx:11-25`)
   - `Badge` variants (`soft|lime|warn|danger|info` + accents
     `sprout|stigma|growth|pistil|haze` and their `-brutal`/`-glass` forms,
     `packages/ui/src/components/ui/badge.tsx:8-49`). `rounded-sm` is **10px** here
     — THE default corner (`packages/design/src/tokens/radius.ts:18-19`), not
     Tailwind's 2px. Never write a hex/hsl in a component; use `bg-primary`
     (= `--color-sprout`), `bg-surface-raised`, `text-sprout`, etc.

3. **The barrel is incomplete — import by subpath.** `@greenroom/ui`
   (`packages/ui/src/components/index.ts`) re-exports only ~30 of 56 components.
   `Drawer`, `Sheet`, `Tabs`, `Select`, `Checkbox`, `RadioGroup`, `Slider`,
   `Switch`, `Skeleton`, `Sidebar`, `Avatar`, `VideoPlayer`, `StatusPage`,
   `ThemeToggle`, `BrandIcon`, and the `file-preview/*` set are reachable **only**
   via `@greenroom/ui/components/<name>` subpaths. Every "exists" mapping below
   names the subpath.

**The "Sprout stays invisible" rule, enforced visually.** Inside a brand portal
the only Sprout mark is a "Powered by Sprout" line in the footer. The Sprout
wordmark (`Logo`, which reads `platformConfig.brand.name` at build time,
`packages/ui/src/components/ui/logo/logo.tsx:2,89`) is **forbidden** in the
Portal shell — the shell renders the **new runtime `<BrandLogo>`** (the org's
uploaded logo from `portal_config.logo_ref`) everywhere a logo appears. `Logo`
appears **only** in Hub and Admin chrome (the two Sprout surfaces) and in the
footer credit. This is a structural guarantee from
[03](./03-app-structure.md#why-one-app-not-several), surfaced here as a UI rule:
a Portal screen never shows the Sprout green-wordmark; it shows the brand's mark
skinned by `<BrandStyle>`.

## Surface 1 — The Hub (the one Sprout-branded surface)

Route group `hub/*` ([03 route map](./03-app-structure.md#route-map)), rendered
at the bare `sproutportal.ca` apex where `RouterContext.brand` is `null`. This
is the **only** screen that uses the Sprout `Logo` wordmark and `PlatformNav`
chrome.

**Layout.** A `Sidebar` (`@greenroom/ui/components/sidebar`, cookie-persisted
collapsible, mobile `Sheet` fallback) left-rail: Your Portals · Leaderboard ·
Education Award · Notifications. Main column is a single scroll of five blocks
matching the spec's Hub inventory:

- **(a) Your Portals** — a responsive grid (`grid-cols-1 sm:grid-cols-2
lg:grid-cols-3`) of `Card variant="default"` tiles, one per brand the user is a
  member of (from the guestlist `member` rows joined to `org_brand_directory` —
  the slug/name/logo mirror, refreshed by a **guestlist org-hook push**
  (`syncOrgDirectory`) with an **hourly cron reconcile** as the self-healing
  backstop, [02 §1](./02-data-model.md#1-tenancy--brand-config)). The mirror is
  display-only: a brand tile's name/logo may briefly be stale, but a tile never
  leaks another brand's data because the portal render always derives `brand_id`
  from the resolved org, never from the mirror row. Each tile = `<BrandLogo>`
  - brand name + an unread-count `Badge variant="danger"` driven by
    `notifications` where `read_at IS NULL` ([02 §11](./02-data-model.md#11-hub--gamification)).
    Below, a **"Brands you can join"** sub-grid: tiles with a `Button` "Request
    Access" that calls `requestAccess` (`hub.functions.ts`), writing a
    `portal_access_requests` row; the button flips to a disabled `Badge
variant="soft"` "Requested" optimistically.
- **(b) Leaderboard** — a bordered table (identity's admin-table pattern,
  `type-page-title` header + `type-mono-label` th,
  `workers/identity/src/routes/_dashboard/admin/orgs/index.tsx:19,36`) of the
  platform-wide top-5 by composite score, reading the `user_brand_scores`
  snapshot **summed across the user's brands for the current period**
  ([02 §11](./02-data-model.md#11-hub--gamification)). The composite `score` is
  materialised by the cron per a single `SCORE_WEIGHTS` formula
  (`round(0.55·quizPoints + 0.30·deckPoints + 0.15·activityPoints)`, weights +
  activity coefficients in one const in `jobs/cron.ts`,
  [02 §11](./02-data-model.md#11-hub--gamification)) — this panel only renders the
  already-computed rows; no scoring math lives in the UI. The user's **own rank is
  always pinned** as a sticky last row even when outside the top-5 — a `Card
variant="dark"` (Indica surface, cream ink) with a `type-stat` rank number.
- **(c) Award of the Month** — a hero `Card variant="dark"` reading
  `education_award`: `covers_text`, a **live countdown** to `closes_at` (a new
  `<Countdown>` tick component, `setInterval` 1s, `aria-live="off"` with a
  `role="timer"` label updated per minute to avoid SR spam), a **semi-anonymous
  leader** ("Leader · 412 pts"), and the user's **gap to first**. Framing copy is
  strictly the education-fund language (`fund_description`) — never prize/reward/
  cash (product law).
- **(d) Last Month's Winner** — a small `Card` with `winner_name` + the period,
  reading the **prior closed period's** `user_brand_scores` row (the Education
  Award winner is the prior month's top score, not the live current-period board).
- **(e) Notification settings** — `/hub/notifications`: per-brand, per-type
  switches via `useAppForm` `SwitchField`, persisting `notification_prefs` rows.
  **There is no single global switch** (product rule); the UI offers only the
  granular grid.

**Mobile.** Sidebar collapses to the `Sheet` drawer (hamburger). The five blocks
stack; Your Portals → single column; the leaderboard table switches to a stacked
`Item` list (`@greenroom/ui/components/item`) so columns don't overflow.

**Interaction/motion.** Award countdown ticks; the unread badge **polls** in v1
— a lightweight `getUnreadCounts` GET on Hub mount + on `window` focus + a 30s
interval, `sessionStorage`-seeded to avoid a flash — and animates a subtle
**pulse on increment** (fired when a poll returns a higher count). The DO push
channel stays reserved for in-section real-time (chat/feed) where it already
exists; cross-brand push for the Hub badge is deferred to P7 (mobile wrapper +
push provider, riding `notification_prefs`). (Default — add a per-user push
channel pre-mobile only if the operator wants instant cross-brand badge updates.)
Tapping a portal tile navigates to that `<slug>.sproutportal.ca` host (a real
cross-host navigation — leaving the Hub, the one place we route between hosts).
Every brand subdomain resolves through a **single wildcard custom-domain route
`*.sproutportal.ca` on the bouncer worker** backed by a zone wildcard TLS cert
(Advanced Certificate Manager / Total TLS), so a new brand is a row of data with
**zero per-brand provisioning** — no Cloudflare-for-SaaS custom hostname per
brand ([07 §1.3](./07-deployment.md)).

**Empty/loading/error.** No memberships → an empty-state `Card` "Join your first
brand" pointing at the join sub-grid. Loading → `Skeleton`
(`@greenroom/ui/components/skeleton`) rows for the leaderboard + tile placeholders.
Errors → `StatusPage` (`@greenroom/ui/components/status-page`, `error` variant).

**Accessibility.** Sidebar ships keyboard shortcut + focus management already.
Countdown uses `role="timer"`; rank table has a `<caption>`; "Request Access"
buttons carry `aria-label` including the brand name.

## Surface 2 — Landing screen (per brand portal)

Route `_portal/index.tsx` inside the persistent shell `_portal.tsx`
([03](./03-app-structure.md#the-one-page-shell--section-layer-system)). First
view each session per brand. Three persistent pieces mount here and **never
unmount** while section layers open/close.

**Layout (desktop).** The `RotatingHero` fills the viewport center: brand images
(`hero_slides.image_ref`) behind `<BrandLogo>` + the brand `tagline` + exactly
**one** "Enter Portal" `Button` (the spec's single CTA). `BannerRail` flanks it
as **side columns** (left/right), each column a stack of `banner_cards`. The
`AiBubble` sits **bottom-right**, persistent, with an unread `Badge`.

**Layout (mobile).** Hero goes full-bleed; `BannerRail` collapses to a
**horizontal top strip** above the hero (the spec's responsive note: side columns
desktop → top strip mobile); the "Enter Portal" button is thumb-reachable at the
hero's lower third; `AiBubble` stays bottom-right.

**Components.**

- **`RotatingHero` — BUILD NEW.** No carousel/embla/scroll-snap primitive exists
  in `packages/ui` (grep-confirmed). Build a scroll-snap (or embla) carousel: one
  slide per `hero_slides` row, each slide a brand image with its `category` tag
  rendered as a `Badge variant="*-glass"` overlay, **arrows + dots**, and
  **auto-advance** (~6s, pause on hover/focus, respect `prefers-reduced-motion`
  by disabling auto-advance). The logo/tagline/"Enter Portal" stack is layered
  above the image with a scrim for contrast.
- **`BannerRail` + banner card — COMPOSE.** Each card composes `Card
variant="soft"` + `interactiveMaterials.brutal` (hover-lift). Card body =
  `category_tag` (`Badge`), `headline`, one `line`, and a link into a section
  (`link_json` → `openLayer(section, item)` — **never** an external URL; the
  platform links nowhere out). Dismissible cards show an X that writes a
  `banner_dismissals` row. `liveFrom`/`expiresAt` gate visibility client+server.
  **Impressions** fire `banner_impression` on first paint (IntersectionObserver,
  which must be built — none exists); **clickthroughs** fire `banner_click`.
- **`AiBubble` — BUILD NEW** (detailed under Surface 9).

**Interaction/motion.** Hero auto-advances and keeps its timer because
`_portal.tsx` never unmounts when a layer opens (the core shell guarantee,
[03 step 5](./03-app-structure.md#how-it-maps-onto-tanstack-router)). Dots reflect
the active slide; arrows wrap. "Enter Portal" navigates to `_portal/home` (a
same-shell search/route change — hero/banners/bubble persist).

**Empty/loading/error.** No hero slides configured → a single brand-color
gradient panel with `<BrandLogo>` + tagline (graceful, never a broken carousel).
Loading → the SSR'd brand config means the hero paints with correct brand colors
immediately; image lazy-load shows a `Skeleton` shimmer per slide. Error
resolving brand → `StatusPage` `not-found` ("This portal isn't available").

**Accessibility.** Carousel is an `aria-roledescription="carousel"` region with
slide `aria-label`s ("Slide 2 of 5"); auto-advance pauses on focus-within; arrows
and dots are real `<button>`s with labels; the single "Enter Portal" CTA is the
first tab stop after the logo.

## Surface 3 — Section grid + Drop Sheet entry

Route `_portal/home.tsx`. After "Enter Portal." Six numbered cards + the Drop
Sheet below; banners + AI persist (shell stays mounted).

**Layout (desktop).** A 6-up grid (`grid-cols-2 lg:grid-cols-3`) of section
tiles, each `Card variant="default"` + `interactiveMaterials.brutal`: a number
chip ("01"–"06") as a `Badge`, an icon, a title, a one-line description. Cards
map to the section keys from `portal_config.sections_json`
([02 §1](./02-data-model.md#1-tenancy--brand-config)) — disabled/reordered per
brand:

| #   | Title                                  | Layer (search param) |
| --- | -------------------------------------- | -------------------- |
| 01  | Store Assets                           | `?section=assets`    |
| 02  | PK Decks                               | `?section=decks`     |
| 03  | Quizzes                                | `?section=quizzes`   |
| 04  | Enter the Grow (or brand `feed_label`) | `?section=feed`      |
| 05  | Group Chat                             | `?section=chat`      |
| 06  | Contact                                | `?section=contact`   |

The **Drop Sheet sits below the grid** (Surface 4). The grid is the scroll anchor
that the browser preserves when a layer closes.

**Layout (mobile).** Grid → 2 columns then 1 on narrow; Drop Sheet below scrolls
naturally.

**Components.** Tiles = `Card` + `surfaceMaterials`/`interactiveMaterials` (no new
primitive). The card "04" title reads the brand's renameable
`feed_label`.

**Interaction/motion.** Tapping a tile calls `openLayer(key, …)` — flips the
`?section=` search param; `<LayerStack/>` mounts the matching `SectionLayer`
(`fixed inset-0 z-50` + backdrop-blur) over the grid. Disabled sections render
the tile dimmed/non-interactive. Closing any layer restores the grid's exact
scroll offset (browser-preserved + `PortalShell` `scrollY` ref fallback).

**Empty/loading/error.** A brand with all-but-one section disabled simply shows
fewer tiles. The grid never blocks on section data (each layer lazy-loads on
open). Errors are per-layer, not grid-level.

**Accessibility.** Tiles are `<button>`/`<a>` with the number announced as part
of the label ("Section 2, PK Decks"); the grid is a `role="list"`.

## Surface 4 — Drop Sheet (`// CURRENT LINEUP`)

Below the grid on `_portal/home`. The product + review + rotation hub. Reads
`products` / `drops` / `reviews` ([02 §2](./02-data-model.md#2-products--drop-sheet)).

**Layout (desktop).** A label row "`// CURRENT LINEUP`" (`type-mono-label`), then
**category card rows** — one row per category (Flower · Pre-Roll · Infused · Hash
· Limited), each row a horizontally-scrollable strip of product `Card`s. Limited
cards carry the `available_note` ("when available"). Each card shows name, THC%
(`thc_pct`), format, and a couple of terpenes (`Badge` chips), plus a **"View
Product Details →"** affordance.

**Per-product detail.** Opening a product (deep-linkable via `?item=<productId>`)
expands a detail panel: THC/CBD, terpenes, effects, talking points, format,
batch, plus the **reviews block**. A **"Full PK →"** link jumps to the PK Decks
layer for `product.deck_id` (`openLayer("decks", deckId)`).

- **Desktop detail** = a right-side `Sheet` (`@greenroom/ui/components/sheet`,
  base-ui dialog with backdrop blur) or an in-flow expansion.
- **Mobile detail** = a **bottom `Drawer`** — the `Drawer` primitive already ships
  the bottom-sheet variant (`data-[vaul-drawer-direction=bottom]`, drag handle,
  `max-h-[80vh]`, snap, `packages/ui/src/components/ui/drawer.tsx:51,56`). Use it
  directly; no new primitive.

**Reviews block (lives here).** 1–5 stars + up to 300 chars,
**one per budtender per product** (the `reviews` UNIQUE
`(brand_id, product_id, user_id)`, [02 §2.3](./02-data-model.md#23-reviews--hard-delete-by-compliance)),
showing `author_name` / `store` / date + the running average. Components:

- **`StarRating` — BUILD NEW** (a 1–5 radiogroup of star buttons; no star widget
  exists). Used both read-only (display) and as a form field.
- **Review composer** = `useAppForm` with the `StarRating` field +
  `TextareaField` (300-char counter, `maxLength` + arktype `onBlur`). Budtenders
  **edit/delete their own** (the composer pre-fills the user's existing review;
  optimistic in-place replace per [03 data-loading](./03-app-structure.md#state--data-loading-strategy)).
- Admin **delete** (never edit, never hide) is a separate hard-delete path
  surfaced only under `/admin/reviews` — there is deliberately **no hide toggle**
  in the UI anywhere; credibility is the feature.

**Interaction/motion.** Category strips scroll-snap horizontally. New drops /
limited releases surface here first (a `drops`-driven "NEW DROP" `Badge
variant="lime"`). Posting a review optimistically prepends it and bumps the
average; failure rolls back with a toast.

**Empty/loading/error.** Empty category → a quiet "No products in this category
yet." Loading → `Skeleton` product cards. No reviews → "Be the first to review."
Over-limit body → inline field error before submit (arktype).

**Accessibility.** `StarRating` is an `aria-label`'d radiogroup operable with
arrow keys; the average is announced; the detail Drawer/Sheet traps focus and
restores it to the originating card on close.

## Surface 5 — PK Decks (library + flip-viewer)

`?section=decks`. Reads `decks` / `deck_progress`
([02 §3](./02-data-model.md#3-pk-decks)). PDFs + thumbnails live in roadie R2;
D1 holds only refs + auto-derived `page_count` / `cover_thumb_ref`.

**Layout — deck library.** A grid of deck `Card`s: `cover_thumb_ref` thumbnail
(via `roadie.getReadUrl`), `title`, `product_line`, **page count**, date. Desktop
3-up, mobile 1–2-up. `page_count` + `cover_thumb_ref` are populated
**asynchronously** by the `deck.derive` queue job on finalize (server-side:
**`unpdf`** reads page count + extracts text for the AI corpus, **Cloudflare
Browser Rendering** screenshots page 1 → roadie for the cover thumbnail); until
the job completes the card shows a `FileIcon` "processing" placeholder.

**Layout — full-screen flip-viewer (BUILD NEW).** No flip/PDF-page primitive
exists. Build a `fixed inset-0 z-50` viewer rendered inside the section layer:

- **Page area** — renders the current PDF page **client-side with `pdfjs-dist`
  (pdf.js)**, fetching the inline `getReadUrl({ disposition: "inline" })` PDF and
  rasterising page N to canvas on demand (gives pinch/zoom, no Worker PDF runtime).
  Desktop: **arrows + click-zones** (left third = prev, right third = next).
  Mobile: **swipe** between pages.
- **Page counter** — "3/12" (`type-mono-label`), bound to current page /
  `deck.page_count`.
- **Zoom** — **pinch + double-tap** on mobile; a zoom control on desktop.
- **Thumbnail filmstrip** — a horizontal strip of page thumbnails to **jump** to
  any page; the active page is highlighted.
- **Download button** — shown **only if `deck.download_allowed`**; calls
  `getReadUrl({ disposition: "attachment" })`.

**Interaction/motion.** Page turns animate (slide/flip; respect
`prefers-reduced-motion`). The viewer tracks **flip depth**: it accumulates dwell
per page and records `last_page` + `time_spent_seconds` to `deck_progress`
(upserted) and emits `deck_open` / `deck_flip` / `deck_download` analytics events
([02 §12](./02-data-model.md#12-analytics-events)) — the analytics signal. Replace
= new PDF, same listing (no field-by-field rebuild); the viewer just re-reads the
new `pdf_ref` and `page_count`.

**Empty/loading/error.** No decks → empty-state card. PDF still rendering →
`Skeleton` page + spinner; thumbnail not yet generated (queue lag) → a generic
`FileIcon` placeholder (`@greenroom/ui/components`). Fetch failure →
`StatusPage`-style inline error inside the viewer with a retry.

**Accessibility.** Arrows/dots/filmstrip are real buttons; the page counter is
`aria-live="polite"`; the viewer is a focus-trapped `role="dialog"`
`aria-label="<deck title> deck viewer"`; Escape closes and restores scroll;
keyboard `←/→` page, `+/-` zoom.

## Surface 6 — Store Assets (library + request physical + My Requests)

`?section=assets`. Reads `assets` / `physical_requests`
([02 §4](./02-data-model.md#4-store-assets--physical-requests)).

**Layout — library.** A categorised, **searchable** grid of asset `Card`s, each:
thumbnail (`thumb_ref`), `name`, `category` (`Badge`), `type`, `size`. A search
`Input` + category filter `Tabs`/`Select` head the layer. Desktop 3–4-up, mobile
1–2-up.

**In-platform viewers (everything opens in-platform — nothing downloads to open).**
Per `assets.type`:

- **PDF** → the same full-screen viewer substrate as PK Decks (zoom).
- **image** → a **lightbox — BUILD NEW** (no lightbox primitive exists;
  `file-preview/modal-preview` and `media-preview`,
  `@greenroom/ui/components/file-preview/*`, are the closest substrate to compose).
- **video** → `VideoPlayer` (`@greenroom/ui/components/video-player`, native
  controls + fullscreen + download).
- **zip** → downloads directly (no viewer).

**Two actions per asset.** **DOWNLOAD** (digital, increments `download_count` +
emits `asset_download`) and **REQUEST PHYSICAL** (only when
`physical_available`).

**Request-Physical form.** A `useAppForm` form (in a `Sheet` desktop / `Drawer`
mobile): `quantity` (capped by `physical_max_qty`, `SliderField`/`TextField`),
`store` (pre-filled from roster, read-only `TextField`), shipping address
(`ship_street`/`ship_city`/`ship_province`/`ship_postal` — `province` a
`SelectField`), `contact_name` + `contact_phone`, optional `note`
(`TextareaField`). Submit writes a `physical_requests` row (status `Requested`) →
the brand fulfilment queue, and emits `physical_request`.

**My Requests.** A status list of the user's `physical_requests`: each row an
`Item` with a status `Badge` tracking **Requested → Approved → Shipped**
(or `Declined` with reason); tracking number shown on Shipped. It is a **real
deep-linkable route** at `/requests` (file home `_portal/requests.tsx` in the
shell subtree, [03 route map](./03-app-structure.md#route-map)) so
fulfilment-status notifications can deep-link straight to it; it also surfaces as
a sub-view of the Store Assets layer.

**Interaction/motion.** Download shows a progress toast; request submit confirms
inline and the form collapses to a "Requested ✓" state.

**Empty/loading/error.** No assets in category → empty card. No requests yet →
"You haven't requested anything yet." Loading → `Skeleton` grid. Upload-still-
processing thumbnails → `FileIcon` placeholder.

**Accessibility.** Search is a labelled `Input` with results count announced;
viewers are focus-trapped dialogs; the address form groups fields with a
`<fieldset>`/legend; status badges include text, not color alone.

## Surface 7 — Quizzes (full-screen overlay)

`?section=quizzes`. Reads `quizzes` / `questions` / `question_options` /
`attempts` / `attempt_answers` / `certifications`
([02 §5](./02-data-model.md#5-quizzes)).

**Layout.** A **full-screen overlay — BUILD NEW shell**: an intro → active →
review → result phase machine hosted inside a `fixed inset-0 z-50`
`SectionLayer` (backdrop-blur, like the alert-dialog/sheet overlay pattern).

**Components / five question types.** Each question type is a renderer composed
from existing form primitives (all via `useAppForm` field group, autosaving to
`attempts.answers_json`):

- **multiple_choice** → `RadioField` group.
- **select_all** → `CheckboxGroupField`.
- **true_false** → a two-option `RadioField`.
- **image** (strain-ID from photo) → the question image (`questions.image_ref`
  via roadie) above a `RadioField` of options.
- **matching** (terpene → effect) → a **BUILD NEW** matching widget (drag-to-pair
  or paired `Select`s; no matching primitive exists), reading the right-side
  values from `question_options.config_json`.

**Flow chrome.** A **progress bar** (`question N of M`), **back-stepping**
(prev/next), a **review screen** before submit (a list of answered/unanswered),
then **results**: score, pass/fail vs `pass_threshold`, **per-question
breakdown** (wrong answers reveal the correct answer + the Brand-Admin
`explanation`), retake (if `retakes_allowed` / `max_attempts`). A **certification
quiz** (`cert_name` set) unlocks a named badge **instantly** on the profile
(`certifications` insert + `cert_awarded` event). Quiz results feed the brand
leaderboard, the Hub composite score, and the Education Award race.

**Brand Leaderboard sub-tab.** The spec's Surface 7 in-portal brand-level quiz
rankings render as a **tab/segment inside this same Quizzes layer** — a
`Tabs` strip `Quizzes` | `Leaderboard` at the top of the layer — **no new
route** (stays within the one-page shell, INV-7). The Leaderboard tab shows the
**same bordered top-N table** the Hub uses (identity admin-table pattern,
`type-page-title` + `type-mono-label` th), brand-scoped: it calls the **existing**
`getLeaderboard({ brandId: activeOrgId, period })`
([05 §1.11](./05-api-and-integrations.md#111-hubfunctionsts--hub--leaderboard--notifications)),
which reads `user_brand_scores` via the `user_brand_scores_leaderboard_idx`
`(brand_id, period, score)` index — **no new data, no new server fn, no schema
change**. The budtender's **own rank is pinned** as a sticky row (like the Hub)
and a **period selector** (`SelectField`, calendar-month `YYYY-MM`) heads the
table. The composite score itself is materialised by the cron per the single
`SCORE_WEIGHTS` formula in `jobs/cron.ts`
([02 §11](./02-data-model.md#11-hub--gamification)); this panel only renders the
already-computed rows. Isolation: a brand-B budtender never sees brand-A scores
because `brandId` is the membership-validated `activeOrgId`, not client input
([05 §1.11](./05-api-and-integrations.md#111-hubfunctionsts--hub--leaderboard--notifications)).

**Interaction/motion.** Autosave/resume mid-quiz (the `attempts.answers_json`
buffer + `current_question` resume cursor) — re-opening the layer resumes where
left off. Timed quizzes (`time_limit_seconds`) show a countdown.

**Empty/loading/error.** No quizzes → empty card. Resume banner if an `open`
attempt exists. Network blip mid-quiz → autosave means no answer loss; a "saving…"
indicator. Submit failure → retry without losing answers.

**Accessibility.** Progress is `role="progressbar"` with `aria-valuenow`; the
overlay is a focus-trapped dialog; matching widget is keyboard-operable (not
drag-only); the per-question breakdown marks correct/incorrect with text + icon,
not color alone; the timer is `role="timer"`.

## Surface 8 — Media Feed ("Enter the Grow")

`?section=feed`. The signature feature; brand-renameable via
`portal_config.feed_label`. Reads `posts` / `post_media` / `post_likes` /

Real-time over the single `GroupChatRoom` DO, addressed per post via

no per-cell join). Cell layout reuses the same real-time comment/chat row
component as Group Chat (component inventory) + `VideoPlayer` /
`file-preview/media-preview` for media. Infinite scroll needs a new
IntersectionObserver loader.

**Layout — expanded post overlay.** Tapping a cell expands it:

- **Mobile** → full-screen.
- **Desktop** → a **full-height overlay** (right-anchored `Sheet` or `fixed`

  It shows the **full carousel** (swipe + dots — reuses the `RotatingHero`

  `?item=<productId>` on the home route, [03](./03-app-structure.md#how-it-maps-onto-tanstack-router);
  **not** the `?section=decks` layer). From the product detail the `Full PK →`

  "product + PK deck stacked" reads in the layer model: the product detail with the
  PK-deck jump available, not a literal two-panel view.

**Live comments.** Chronological, name/store/time/text, **max 500 chars,
text-only** (the `comments` body cap), **real-time for everyone** via the DO
broadcast, **hearts** on comments (`comment_likes`), authors delete own / admins
delete any (soft-delete `deleted_at`). **Brand replies are visually distinct** —

(500-char counter).

**Interaction/motion.** New comments **insert in real-time** (DO broadcast →

**Empty/loading/error.** Empty feed → "Nothing growing yet." Loading → `Skeleton`
cells. WS disconnect → a quiet "reconnecting" pill; comments fall back to a fetch

**Accessibility.** Each cell is an article with an accessible name; the carousel
in the overlay is labelled; comment insertion uses `aria-live="polite"` on the
comment list (rate-limited); the `Team` marker is text + badge, not color alone;
like buttons are toggle buttons with `aria-pressed`.

try Contact or book a call" (degrades to the human channels, never a dead end).

**Accessibility.** The panel is a labelled dialog; streamed text uses
`aria-live="polite"`; the slot picker is a keyboard-navigable listbox of times;
the bubble button has an `aria-label` including unread count.

## Runtime theming model (from the UI side)

Each portal must look **entirely different** from one codebase. The UI achieves
this without any per-component work because **the design tokens are already CSS
variables**.

**The variable substrate already exists.** `packages/design/generated/css/tokens.css`
emits every brand colour as a `--color-*` custom property on `:root` (light) and
`[data-theme="dark"]` (dark), and `theme.css` maps the shadcn semantic names onto
Sprout tokens — critically **`--color-primary` IS `--color-sprout`**
(`packages/design/src/theme.css:38`). So `bg-primary`/`text-primary`/`ring-primary`
everywhere resolve to the brand green; redefining `--color-sprout` retints all
"primary" UI in one shot. Defaults are the Sprout light-first cream/forest palette
with the 5 accents `sprout/stigma/growth/pistil/haze`, `rounded-sm` = 10px
(`packages/design/src/tokens/radius.ts:18-19`), and Zerove (display) / Switzer
(text) fonts.

**The runtime override — `<BrandStyle>` (a bounded three-role override).**
`resolveBrandForHost(host)` returns the org's slim `BrandRuntime` skin (token
overrides + logo ref + display name,
[02 §1](./02-data-model.md#1-tenancy--brand-config)). `__root.tsx` SSRs a
`<BrandStyle brand={brand}/>` (`components/brand/`) — an inline scoped `<style>`
that **redefines** exactly **three colour roles**: `--color-primary`
(= `--color-sprout`), one secondary `--color-accent`, and
`--color-background`/surfaces — plus optionally `--font-display`/`--font-body`
(with `@font-face` from roadie-hosted faces) and `--radius-*`, on the portal root.
This matches the authored v2 `BrandTheme` shape
`{ modePolicy?, fixedMode?, light?, dark?, radius?, spacing?, fonts? }`
([02 §1](./02-data-model.md#1-tenancy--brand-config)) 1:1.
The **four other named accents are intentionally non-overridable**: `stigma`
(danger), `growth` (success), `pistil` (warn), and `haze` (info) stay **fixed
Sprout semantic/status tokens**, never brand identity — a brand must not be able
to recolour "danger" green, because status colour carries fixed meaning for a11y,
learnability, and WCAG contrast. Because everything downstream reads those
variables, this single block re-skins the entire portal with **no per-component
change** — cream/forest defaults become the brand's palette, 10px rounding becomes
the brand's radius, the fonts become the brand's pairing, while the status accents
read identically across every brand. (Default — full 5-accent control is a future
opt-in gated behind a **Sprout-Admin** flag, not Brand-Admin, accepting that
status semantics would become brand-overridable.)

**FOUC + dark-mode correctness.** `<BrandStyle>` SSRs in `<head>` alongside the
existing `THEME_INIT_SCRIPT`, so there is no flash and no wrong-brand flicker.
Theme is driven by the **`data-theme` attribute** (not a class), so the override
emits **both** a light and a dark block (or values both modes inherit) — the
org's skin applies in either mode.

**The admin live preview rides the same variables.** `/admin/setup`'s live
preview simply mutates these `--color-*`/font variables client-side as the editor
types (reading the draft theme) — no rebuild, no save round-trip — which is
exactly why the preview is instant and faithful.

**Never via build-time brand.** This is the **runtime** mechanism. It must not
route through `packages/config` / `brand.ts` — that system
brands the whole Sprout fork **once at build time** (the Hub, the `Logo`
wordmark) and a rebuild-per-brand is the opposite of "configuration, never code."

**"Sprout stays invisible," visually.** Two UI guarantees:

1. The Portal shell renders the **runtime `<BrandLogo>`** (org logo from
   `portal_config.logo_ref`, falling back to a tinted `BrandIcon`/`LogoIcon`) in
   every logo slot. The build-time Sprout `Logo` wordmark
   (`packages/ui/.../logo/logo.tsx:89` reads `platformConfig.brand.name`) is
   **never** rendered inside a Portal — only in Hub/Admin chrome.
2. The **only** Sprout mark inside a portal is a single "Powered by Sprout" line
   in the portal shell's own plain inline `<footer>` element
   (`workers/sprout/src/routes/_portal.tsx`) — not a shared `PlatformFooter`
   component. The Hub remains the one fully Sprout-branded surface.

## Forms convention (restated, because it is non-negotiable)

Every form on every surface uses `useAppForm` from
`@greenroom/ui/hooks/use-app-form` with `form.AppField` + the lazy field
components and `form.AppForm` + `form.SubmitButton` (the canonical pattern at
`workers/identity/src/components/auth/sign-in-form.tsx`). **Never** hand-roll
`useState` + native inputs — every Sprout form, including every Brand-Admin
editor, uses `useAppForm` (project memory `feedback_repo_form_convention`).
Validation
is arktype `type({...})` on `onBlur`; this covers the review 300-char cap, the
comment 500-char cap, the request-physical address, the contact topic, and every
Brand-Admin editor.
