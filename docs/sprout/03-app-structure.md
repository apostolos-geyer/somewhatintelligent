# 03 ‚Äî App Structure: the single `workers/sprout` TanStack Start app

> **Scope.** The entire Sprout product ‚Äî every per-brand **Portal**, the
> **Brand Admin** dashboard, and the Sprout-branded **Hub** ‚Äî ships as **one**
> TanStack Start app: `workers/sprout`. It owns its **own D1** for domain data and
> leans on the platform spine (guestlist = auth/org, roadie = R2 blobs, promoter
> = email, bouncer = edge routing per brand subdomain). Quiz-taking and group
> chat are two of Sprout's built-in sections, not separate apps.
>
> This document is the structural blueprint for `workers/sprout`: a single
> TanStack Start worker with its own D1 schema, a queue + scheduled worker for
> analytics rollups, and a Durable Object backing real-time group chat and feed
> comments. (Sprout's quiz and group-chat sections trace their design back to
> two prior-art apps, `apps/quiz` and `apps/chat`, which have since been folded
> in and removed from this repo.) Every convention below reflects a real,
> working part of the greenroom platform ‚Äî identity's router and middleware
> patterns, bouncer's wildcard routing, roadie's blob namespacing ‚Äî not an
> invented pattern.

---

## Why one app, not several

The spec's "One engine, infinite skins" rule (one codebase runs MTL Cannabis,
Dom Jackson, Lite Label, Lowkey, Rosin Heads‚Ä¶ ‚Äî per-brand is **data only**) maps
directly onto a single multi-tenant worker. A new brand portal is a row in the
app's own D1 plus an organization in guestlist ‚Äî **never** a new deploy. Splitting
Portal / Admin / Hub across apps would re-introduce cross-origin auth hops and
duplicate the brand-resolution + theming machinery three times. We consolidate.

The product rule **"Sprout stays invisible ‚Äî only the footer credit inside a
portal; the Hub is the one Sprout surface"** is enforced structurally by the
route map below: the Hub lives under `/hub`, every brand Portal renders at the
**host root** of `<brand>.sproutportal.ca`, and the Sprout wordmark
(`@greenroom/config` build-time brand, via `Logo`) is reachable **only** from
Hub/Admin chrome ‚Äî never from a Portal shell, which uses a **new runtime
`<BrandLogo>`** instead.

## The one-page shell + section-layer system

### The product rule

> One page, one shell. Sections are **layers** over it (no routing). Closing a
> layer **restores exact scroll position**. The rotating hero, persistent
> banners, and persistent AI bubble stay mounted while layers open/close.

### How it maps onto TanStack Router

The naive "open a section = navigate to `/section`" approach unmounts the shell
and loses scroll. We avoid that. The shell is a **single pathless layout route**
and section state lives in a **search param + a layer store**, not in the path:

1. **`_portal.tsx` is the persistent shell.** It is a pathless layout route
   (TanStack convention: `_`-prefixed segment, like identity's `_dashboard.tsx`
   / `_auth.tsx`). It renders the `RotatingHero`, `BannerRail`, `AiBubble`, and
   the `<LayerStack/>` **above** an `<Outlet/>`. Because the shell route never
   leaves the matched tree while a section opens, **the hero carousel keeps its
   timer, the banners keep their impressions, and the AI bubble keeps its
   websocket/poll** ‚Äî they are React subtrees that simply never unmount.

2. **A section opens via a search param, not a path change.** The home route
   (`_portal/home.tsx`) owns a typed search schema:

   ```ts
   const homeSearch = type({
     "section?": "'assets'|'decks'|'quizzes'|'feed'|'chat'|'contact'",
     "item?": "string", // deep-link a product / deck / post / thread
   });
   export const Route = createFileRoute("/_portal/home")({
     validateSearch: homeSearch,
     // shell + grid render the same DOM; section is read off search
   });
   ```

   Opening "02 PK Decks" calls
   `navigate({ search: (s) => ({ ...s, section: "decks" }) })`. TanStack treats
   this as a **same-route search change** ‚Äî `_portal.tsx` and `_portal/home.tsx`
   do **not** remount, so the grid stays in the DOM underneath. The
   `<LayerStack/>` reads `Route.useSearch()` and renders the matching
   `SectionLayer` (a `fixed inset-0 z-50` overlay with backdrop) on top.

3. **Scroll restoration is automatic + explicit.** The router is created with
   `scrollRestoration: true` (exactly as `workers/identity/src/router.tsx:12`).
   Because the grid never unmounts when a layer opens, its scroll offset is
   physically preserved by the browser ‚Äî closing the layer
   (`navigate({ search: (s) => ({ ...s, section: undefined, item: undefined }) })`)
   reveals the grid at the **same** offset. For the cases the browser cannot
   preserve (a layer that itself scrolled the body on mobile), `PortalShell`
   snapshots `window.scrollY` into a ref on open and restores it on close. The
   product's "restores exact scroll position" is therefore the **default**
   behavior, not a feature we bolt on.

4. **Why search params over route masks or a pure store.**
   - **Route masks** would let us deep-link `/decks/123` while keeping the shell,
     but masks change the displayed URL without changing the matched route ‚Äî they
     are designed for modals that survive refresh-to-canonical, and they fight
     scroll restoration because the masked route is a _different_ match. We don't
     use them.
   - **A pure client store** (Zustand) gives the cleanest "layer" semantics but
     loses **deep-linkability**: the spec needs banner cards and AI/feed CTAs to
     link _into a section_ (`View Product Details ‚Üí`, `Full PK ‚Üí`), and the Hub's
     unread badges link to a specific post. A URL the shell can SSR is required.
   - **Search params win:** they are part of the same route match (shell stays
     mounted), they are typed + validated (arktype `validateSearch`), they
     survive refresh and are shareable/deep-linkable, and they leave
     `scrollRestoration` intact. A thin `useLayerStack()` hook wraps the
     search-param read/write so components call `openLayer("decks", id)` /
     `closeLayer()` without touching router internals. A modal/overlay **stack**
     (assets ‚Üí request-physical form ‚Üí confirmation) is an array in that hook's
     state, layered with ascending `z-index`; the bottom of the stack is always
     the search-param section so a refresh re-opens the section but collapses
     transient sub-modals ‚Äî the correct UX.

5. **What is mounted once, forever (per portal session):** `RotatingHero`
   (embla/scroll-snap carousel ‚Äî a **NEW** primitive, no carousel exists in
   `packages/ui`), `BannerRail` (side columns desktop / top strip mobile,
   composed from `Card` + `surfaceMaterials`), and `AiBubble` (bottom-right,
   unread badge). They live in `_portal.tsx`, **outside** the `<Outlet/>` and
   **outside** the `<LayerStack/>`, so no section open/close can touch them.

This is the entire "no routing, sections are layers" mechanism: **one pathless
layout route + a typed search param read by a layer-stack hook + the router's
own scroll restoration.**

### Host ‚Üí brand selection

One worker instance serves every brand. Bouncer routes a single **wildcard**
host `*.sproutportal.ca` (and `sproutportal.ca` itself for the Hub) to the
`SPROUT` binding ‚Äî bouncer already supports single-label `*.`-prefixed wildcard
hosts. The chosen wildcard mechanism is a **single wildcard custom-domain route
`*.sproutportal.ca` on the bouncer worker backed by a zone wildcard TLS cert**
(Advanced Certificate Manager / Total TLS for `*.sproutportal.ca`), **not**
Cloudflare for SaaS custom hostnames: every brand is a subdomain of the operator's
own apex `sproutportal.ca`, so one wildcard cert + one route covers all brands with
zero per-brand provisioning ‚Äî exactly the "a new brand is a row of data" goal.
Cloudflare for SaaS is reserved for the future case where a brand brings its own
apex/vanity domain (out of scope now). The app then resolves the org from the
`Host` header:

- **`__root.tsx` `beforeLoad`** reads the incoming host and calls a server-only
  `resolveBrandForHost(host)` (`lib/brand.server.ts`). For a brand subdomain
  (`<slug>.sproutportal.ca`) it resolves the leftmost label ‚Üí org via the
  `org_brand_directory` mirror (slug ‚Üí org_id) and loads that org's **runtime
  brand skin** (logo ref, colours, fonts) from the app's **own D1** (the
  `brand_theme` table plus `portal_config` identity columns, keyed by `org_id`),
  returning it into `RouterContext.brand`; portal content (tagline, feed label,
  section toggles) and banners load in parallel from the portal shell loader,
  and hero slides stream from the landing route. This makes the brand available
  **before first paint**, so the runtime theme `<style>` and `<BrandLogo>` SSR
  correctly (no FOUC, no wrong-brand flash). The directory mirror is kept fresh
  by a **guestlist org-hook push** (authoritative for onboarding latency: a new
  brand's `<slug>.sproutportal.ca` resolves immediately after provisioning) with
  an **hourly reconciliation cron** as the drop-recovery backstop (see 05 ¬ß7.1).
- For the **bare `sproutportal.ca` apex** (and `app.`/`hub.` if used), `brand`
  is `null` and the dispatcher renders the **Hub** ‚Äî the one Sprout-branded
  surface.
- For the **signed-in write path**, the active org comes from the **verified
  envelope** (`context.principal.activeOrgId`), never from the host or input ‚Äî
  the host resolves _which portal you are looking at_; the envelope decides
  _which brand you may write to_.

This is a **NEW runtime mechanism** layered on the org-plugin precedent ‚Äî
`org.metadata` already carries `theme.light/dark["--color-primary"]` tokens,
parsed in `workers/identity/src/routes/_dashboard/admin/orgs/$id.tsx:42-52`. We
extend that idea into first-class per-org `brand_theme` + `portal_config` tables. It is **not**
the build-time `packages/config` brand system (that brands
the whole Sprout fork once).

### Middleware (all in `lib/middleware/auth.ts`)

Built with `createPrincipalGate` from `@greenroom/kit/react-start`, copying
`workers/identity/src/lib/middleware/auth.ts` verbatim, plus one new gate:

- **`envelopeMiddleware`** ‚Äî installed globally in `start.ts`
  (`requestMiddleware: [requestLogger, envelopeMiddleware]`, exactly as
  `workers/identity/src/start.ts:9`) so every gate dedupes the one envelope verify.
- **`requireUserMiddleware`** ‚Äî `predicate: p => p.kind === "user"`; `onReject`
  ‚Üí `redirect({ href: "/sign-in" })`. Gates Hub + Portal write fns.
- **`requireAdminMiddleware`** ‚Äî adds `&& isAdminRole(p.actor.role)` (platform
  admin, god-mode). Gates `/sprout-admin`. **Never** `role === "admin"` ‚Äî
  `user.role` is comma-separated; use `isAdminRole` from `@greenroom/kit/roles`.
- **`requireBrandRole`** ‚Äî **NEW**: gate on `kind === "user"`, then inside the
  handler resolve host ‚Üí org and check `getCallerOrgRole(orgId)` (a guestlist
  round-trip via `org.getActiveMemberRole`) against a pure predicate in
  `policy.server.ts` (`decideBrandAdmin({ actorRole, orgRole })`). Platform admin
  bypasses the org-role check (`isPlatformAdmin`). The BA org-plugin role
  (owner/admin/member) is **separate** from the platform role on the envelope;
  Brand-Admin theme edits gate on the org `theme:['update']` permission.
- **Brand-config injection** is a `RouterContext` value, not middleware: the
  `__root.tsx` `beforeLoad` puts `brand` into context so every route + loader can
  read it without re-resolving the host.

Pure predicates live in `policy.server.ts` (no `cloudflare:workers` import ‚Üí
unit-testable, returning discriminated unions `{ ok:true } | { ok:false, reason }`);
binding-touching helpers live in `runtime.server.ts`.

## Quiz and chat features

Sprout ships two built-in interactive sections ‚Äî quiz-taking and group chat ‚Äî
each with its own schema, its own D1 tables, and brand-scoped tenancy. Both
use Sprout's own schema and server-fn conventions, and both are hosted as
`SectionLayer` overlays inside the one-page shell rather than as standalone
apps or routes.

### Quizzes

- **Data:** doc [02 ¬ß5](./02-data-model.md#5-quizzes) is the canonical,
  authored quiz schema. The portal's quizzes **stand alone** (no
  `courses`/`lessons`/`completions` wrapper): the authored tables are
  `quizzes`, `questions`, `question_options`, `attempts`, `attempt_answers`,
  and `certifications`. Every quiz table carries a nullable indexed
  `brand_id`, plus a denormalized `attempts.brand_id`, so quizzes and attempts
  are tenant-scoped to the owning brand/org. Grading happens server-side and
  questions/answers are redacted before they reach the client. The
  `audit_log` table is authored once for the whole app in
  [02 ¬ß12](./02-data-model.md#12-analytics-events) (every mutation calls
  `writeAudit`), not re-imported per-section.
- **Server fns:** `sprout/src/lib/quizzes.functions.ts` holds the quiz server
  fns (`listQuizzes`, `startAttempt`, `resumeAttempt`, `gradeAttempt`), gated
  by `requireUserMiddleware`, with identity read from `context.principal` and
  brand from `activeOrgId`.
- **UI:** the quiz "take" flow is a full-screen `SectionLayer` overlay
  (`fixed inset-0 z-50`, the spec's full-screen quiz overlay), driven by an
  intro/active/result phase state machine. Question builder admin lives under
  `/admin/content/quizzes`. The certification badge + leaderboard feed the
  Hub's composite learning score and the Education Award race.
- **Collaborators ‚Üí org membership:** the spec's roster is the BA org
  `member` table ‚Äî quiz collaborators are driven by org membership rather
  than a parallel hand-seeded table.

### Group Chat (Durable Object)

- **DO:** `sprout/src/room-server.ts` exports `GroupChatRoom` from
  `sprout/src/worker.ts` (`export { GroupChatRoom }`; binding
  `GROUP_CHAT_ROOM`). It is **one** class, addressed per-room via `idFromName`:
  group chat = `idFromName(brandId)` (one instance per brand); feed
  live-comments = `idFromName(\`${brandId}:${postId}\`)`(one instance per
post). The v1 DO migration is FROZEN at`migrations:[{tag:'v1', new_sqlite_classes:['GroupChatRoom']}]`. The DO
verifies the bouncer envelope **itself** in `onConnect`: it derives
`expectedHost`**per-connection** from the incoming WS-upgrade`Host`header
(not a single static URL, since every brand portal has a different host),
validates it against the`\*.sproutportal.ca`single-label wildcard pattern
(mirroring bouncer's`routes.ts`), resolves the leftmost label ‚Üí org via
`org_brand_directory`, then for authenticated connections asserts the
envelope principal's `activeOrgId === resolved org_id`before admitting the
socket (reject`1008`otherwise). This ties the DO's per-room`brand_id`
  scope to the verified envelope so a member of brand A cannot open brand B's
  room even if a room id leaks.
- **WS interception:** `worker.ts` intercepts `/ws/*` via `routePartykitRequest`
  **before** `startEntry.fetch`, because TSS chokes on 101 upgrade responses.
  The dev-envelope stamper runs before the WS branch so DO upgrades also carry
  an envelope.
- **UI:** `components/sections/chat/` holds the DO-backed room UI (message
  list, composer, presence). The **media feed's** live comments reuse the
  **same single `GroupChatRoom` class** addressed per-post
  (`idFromName(\`${brandId}:${postId}\`)`) ‚Äî the feed is the chat row layout
with media cells. A second class is **not** shipped: a brand chat room and a
post-comment room are the same shape (durable message log in DO SQLite +
presence + hearts), so a second class would duplicate code for zero
behavioural gain and enlarge the irreversible v1 set. `MediaFeedRoom`is
named **only** as a documented future`tag:'v2', new_sqlite_classes:['MediaFeedRoom']` escape hatch if a single
post's comment fan-out ever needs independent hibernation/sharding ‚Äî
additive and allowed, never shipped in v1. Feed-comment real-time is **not**
mirrored to the presence/`chat_rooms`tables (those stay group-chat-only);
the durable comment log is the D1`comments` table, the DO holds ephemeral
  per-post fan-out keyed by post id.

**Roadie namespacing.** References are scoped by `caller_app` **first** (not
by `resourceType` namespacing alone): every blob Sprout mints ‚Äî deck PDFs,
brand logos, recording egress ‚Äî is registered under `caller_app:"sprout"`
from the skeleton onward, so a blob registered under a different caller app
is invisible to Sprout's own lookups (treated as `reference_not_found`). This
keeps tenant data cleanly isolated per app with no cross-app leakage, and
needs no backfill for a fresh fork. A fork carrying production data
registered under a different caller app runs a one-time migration script in
`workers/sprout` that re-registers each legacy blob under `caller_app:"sprout"`
with the appropriate sprout `resourceType` and rewrites the D1 `*_ref`
handles; because roadie dedup is global on content hash, this is a
**metadata-only re-reference, not a byte copy**.

## State / data-loading strategy

- **SSR loaders for the first paint of each surface.** Route `loader`s call GET
  server fns; components read `Route.useLoaderData()` (identity pattern). The
  Portal shell loads brand config + section visibility in `__root`/`_portal`
  `beforeLoad`/`loader` so the hero, banners, and grid render server-side with
  the correct brand.
- **Lazy per-layer loads.** A section layer fetches its own payload when it opens
  (search param flips). The grid never blocks on six sections' worth of data.
- **Real-time via the Durable Object.** Group chat and media-feed comments
  subscribe over the `/ws/*` channel to the DO; the DO broadcasts new
  messages/comments to all connected clients (chronological, real-time for
  everyone). Feed like-counts and comment-counts update live off the same
  broadcast.
- **Optimistic updates for reviews / comments / likes.** Mutations
  (`upsertMyReview`, `addComment`, `likePost`) apply optimistically client-side,
  then reconcile against the server fn result / DO broadcast. Reviews are
  **one per budtender per product**; the optimistic write replaces the user's own
  prior review in place. Admin **delete** (never edit, never hide) is a
  hard-delete server fn gated by `requireBrandRole`; credibility is the feature,
  so there is deliberately no "hide" path.
- **Hub unread badge = poll in v1** (default ‚Äî change if the operator wants
  instant cross-brand badge updates pre-mobile, which means a net-new per-user
  notification DO/SSE channel). A lightweight `getUnreadCounts` GET runs on Hub
  mount + on window-focus + a 30s interval, `sessionStorage`-seeded to avoid a
  flash; the "pulse on increment" fires when a poll returns a higher count. The
  standing preference is push, but the Hub is the apex/cross-brand surface with no
  per-brand DO connection open, so a dedicated push channel for low-urgency unread
  counts is net-new infra; push lands in **P7** when the mobile wrapper + push
  provider arrive (push rides `notification_prefs`). The DO push channel stays
  reserved for in-section real-time (chat/feed) where it already exists.
- **Analytics is write-mostly.** Engagement events (deck flip depth, deck time,
  product views, session join duration, asset downloads, AI questions) write to
  the **Analytics Engine** dataset (`AE` binding) and/or D1 rollup tables; the
  `/admin/analytics` loaders read aggregates; `exportCsv` streams from D1. The
  `queue`/`cron` jobs do the heavy rollups (leaderboard recompute, award
  countdown, most-missed-question) off the request path.

## Theming: runtime brand-config tokens

Each portal must look **entirely different** from one app. The design system is
already built for this: every brand colour is emitted as a `--color-*` CSS custom
property on `:root` (light) and `[data-theme="dark"]`
(`packages/design/generated/css/tokens.css`), and `theme.css` maps the shadcn
semantic names (`--color-primary`, `--color-background`, ‚Ä¶) onto Sprout tokens
(`--color-primary` **is** `--color-sprout`). So re-skinning is **redefining CSS
variables**, not rebuilding components.

**Mechanism ‚Äî a runtime `<BrandStyle>` injected in `__root.tsx` head:**

1. `resolveBrandForHost(host)` (server-only) returns the org's slim
   `BrandRuntime` skin: a small map of token overrides
   (`--color-primary`, `--color-background`, `--color-accent`, optional
   `--font-display`/`--font-body`, optional `--radius-*`) from `brand_theme`,
   plus a logo asset ref and display name.
2. `__root.tsx` renders a `<BrandStyle brand={brand}/>` (`components/brand/`) that
   SSRs an inline scoped `<style>` redefining those `--color-*` vars on the
   portal root. Because the design tokens are already variables, this single
   `<style>` retints **all** `bg-primary`/`text-primary`/`ring-primary` UI in one
   shot ‚Äî no per-component work. It SSRs alongside the existing
   `THEME_INIT_SCRIPT` so there is **no FOUC** and no wrong-brand flash. A brand
   retints exactly **three** roles ‚Äî `--color-primary` (= `--color-sprout`), one
   secondary `--color-accent`, and `--color-background`/surfaces ‚Äî plus optional
   `--font-display`/`--font-body` and `--radius-*`. The four other named accents
   (`stigma`=danger, `growth`=success, `pistil`=warn, `haze`=info) stay **fixed
   Sprout semantic/status tokens, intentionally non-overridable** by Brand-Admin,
   so danger reads as danger across every brand for a11y/WCAG contrast. (Full
   5-accent control is a future opt-in gated behind a _Sprout_-Admin flag, not
   Brand-Admin.)
3. It must stay compatible with `data-theme` light/dark switching: the override
   emits **both** a light and a dark block (or variable values both modes
   inherit), so an org's skin applies in either mode ‚Äî theme is driven by the
   `data-theme` attribute, not a class.
4. **Logo/wordmark is the one build-time-bound component.** `Logo`
   (`packages/ui/.../logo/logo.tsx`) reads `platformConfig.brand.name` (the
   Sprout fork brand) ‚Äî correct for Hub/Admin chrome, **wrong** for a brand
   portal. So `components/brand/BrandLogo` is **NEW**: it renders the org's
   uploaded logo (a roadie R2 url via `getReadUrl`) and falls back to `BrandIcon`
   / `LogoIcon` tinted with the org's colours. The Portal shell uses `<BrandLogo>`
   exclusively; `Logo` (Sprout wordmark) appears only in the Hub and the
   "Powered by Sprout" footer credit.

**Fonts** come from the spec's curated font-pairing list; an org's chosen pair
sets `--font-display`/`--font-body` in the `<BrandStyle>` block (with
`@font-face` from roadie-hosted font files where a brand ships custom faces).

**Do not** route any of this through `packages/config` ‚Äî
that brands the whole fork (Sprout) at **build time**; the per-org skin is a
**runtime** read from D1, edited live via `/admin/setup` (with the spec's LIVE
PREVIEW, which simply mutates the same CSS variables client-side before save).

## Resolved decisions

The choices this structural plan surfaces are settled as follows (decision +
rationale; `(default ‚Äî change if X)` marks a default that can be revisited):

- **One DO class, not two.** Ship exactly **one** class `GroupChatRoom` (binding
  `GROUP_CHAT_ROOM`); the v1 migration is FROZEN at
  `migrations:[{tag:'v1', new_sqlite_classes:['GroupChatRoom']}]`. Group chat =
  `idFromName(brandId)`; feed live-comments = `idFromName(\`${brandId}:${postId}\`)`.
A brand chat room and a post-comment room are the same shape (durable message
log in DO SQLite + presence + hearts), so a second class duplicates code for
zero behavioural gain and enlarges the irreversible v1 set. `MediaFeedRoom`is a
documented future`tag:'v2'` additive escape hatch only, never shipped in v1.
- **Wildcard mechanism = wildcard route + zone wildcard cert** (default ‚Äî add
  Cloudflare for SaaS for a single brand only if it later needs its own apex
  domain). `*.sproutportal.ca` is one wildcard custom-domain route on the bouncer
  worker backed by an ACM / Total TLS zone wildcard cert; every brand is a
  subdomain of the operator's own apex, so one cert + one route covers all brands
  with zero per-brand provisioning. Cloudflare for SaaS is reserved for the future
  customer-owned-vanity-domain case (out of scope now).
- **Per-request DO `expectedHost`.** Under wildcard brand subdomains the DO derives
  `expectedHost` **per-connection** from the WS-upgrade `Host` header, validates
  the `*.sproutportal.ca` single-label shape, resolves the leftmost label ‚Üí org via
  `org_brand_directory`, then asserts `principal.activeOrgId === resolved org_id`
  before admitting an authenticated socket (reject `1008` otherwise). Accepting any
  `*.sproutportal.ca` alone would permit cross-brand room access, so host +
  `activeOrgId` must both gate ‚Äî tenant isolation makes this non-negotiable.
- **Roadie blob backfill** (default for greenfield forks: **NO-OP**). All sprout
  blobs are minted under `caller_app:"sprout"` from the skeleton onward; a
  data-carrying fork runs a one-time re-register-under-`sprout` backfill
  (metadata-only re-reference, not a byte copy), gated on the 09 ¬ß8 "does any fork
  carry existing quiz/chat rows?" prerequisite.
- **Hub unread badge = poll in v1** (default ‚Äî change if the operator wants instant
  cross-brand badge updates pre-mobile, which means a net-new per-user notification
  DO/SSE channel). `getUnreadCounts` on mount + window-focus + 30s interval,
  `sessionStorage`-seeded; push lands in **P7** when the mobile wrapper + push
  provider arrive (push rides `notification_prefs`).
- **Brand config lives in dedicated app-D1 tables** (`brand_theme` +
  `portal_config`) keyed by `org_id`, **not**
  the guestlist `org.metadata` theme-token JSON precedent. Dedicated tables hold
  the richer theme/section/feed-label data the portal needs; they extend (not replace)
  the org-plugin precedent that already carries `theme.light/dark` tokens. The
  `org_brand_directory` mirror (slug ‚Üí org_id) is refreshed by a guestlist
  org-hook push (authoritative for onboarding latency) with an hourly reconcile
  cron as the drop-recovery backstop.
- **AI generation = Workers AI** (default ‚Äî the `generate()` seam in `lib/ai.ts`
  makes swapping to an external LLM a one-file change if eval quality demands it):
  `@cf/meta/llama-3.1-8b-instruct` for generation, `@cf/baai/bge-base-en-v1.5`
  (768-dim) for embeddings ‚Üí `VECTORIZE` index dimension 768. No external-LLM
  secret is provisioned for v1.
- **Real-time = RealtimeKit Core SDK** (client) **+ RealtimeKit REST** (server, to
  mint meeting/session + auth tokens), not the raw SFU push/pull-tracks API
  (default ‚Äî RealtimeKit gives managed participant/recording lifecycle out of the
  box). Managed recording egress targets the project R2 bucket; the
  recording-complete webhook registers the object with roadie under
  `caller_app:"sprout"`. RealtimeKit app id + secret are new `provided` wrangler
  secrets scoped to `['sprout']`.
- **Admin charting = build-new SVG primitives** (`BarChart` + `Sparkline` +
  `TopNBars`), no chart library (default ‚Äî adopt `@observablehq/plot` or `visx`
  later only if rich interactive/exportable time-series charts are required).
  Per-budtender matrices and most-missed/top-AI-question rollups render as the
  existing identity admin TABLE.
- **Admin reorder = one keyboard-first `SortableList` primitive**: move-up/move-down
  buttons are the mandatory a11y baseline (writing `order_idx` / array order), with
  `@dnd-kit/core` + `@dnd-kit/sortable` pointer drag as a progressive enhancement
  (default ‚Äî ship buttons-only and skip the `@dnd-kit` drag layer if zero new deps
  is required; drag is enhancement-only). Reused for `hero_slides.order_idx`,
  `portal_config.sections_json` order, and `banner_cards.order_idx`.
